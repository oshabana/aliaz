# Security and Privacy Model

Aliaz is designed for local-first alias management with optional encrypted sync.
The useful version of the promise is simple: the sync server should be able to
store and move your aliases between machines without being able to read them.

This document explains what is protected, what the server can still see, and
which assumptions have to hold for the model to work.

## Security goals

Aliaz aims to make these statements true:

- Alias names and commands are encrypted before they leave your computer.
- The recovery phrase is never sent to the sync server.
- A database-only compromise of the hosted sync service should not reveal alias
  contents.
- A new computer can recover aliases only with both the account password and the
  recovery phrase.
- Local-only use does not require an account or any network service.

Aliaz does not claim that metadata is hidden, that local files are encrypted at
rest, or that the project has been formally audited.

## What happens during registration

When you run:

```sh
aliaz register --username ada
```

Aliaz prompts for an account password, creates a recovery phrase locally, stores
that phrase in the operating system credential store, and prints it once.

The sync server receives the username and password for account creation. The
server stores a password hash, not the plaintext password.

The recovery phrase stays on the client. It is not sent to the sync server. Save
it somewhere safe: Aliaz cannot recover encrypted aliases without it.

## What happens on a new computer

When you run:

```sh
aliaz login --username ada
aliaz sync
```

Aliaz prompts for two secrets:

- The account password, used to authenticate to the sync server.
- The recovery phrase, used locally to decrypt aliases.

The login request sends the username and password to the sync server. The
recovery phrase is validated and stored locally. During sync, the client
downloads encrypted records, derives a local encryption key from the recovery
phrase, and decrypts the aliases on the new computer.

This means the account password and recovery phrase have different jobs. The
password proves you are allowed to access the account. The recovery phrase is
what makes the encrypted alias data readable.

## Logging out on one machine

When you run:

```sh
aliaz logout
```

Aliaz removes the local sync configuration and deletes the locally stored
recovery phrase for the configured account. This disconnects the current machine
from sync until you run `aliaz login` again.

Logout does not delete the local alias database, the generated shell integration
files, or server-side account data.

## What is encrypted

Synced alias records are encrypted on the client before upload. The encrypted
payload includes the alias name, command, deletion state, and update timestamp.

Aliaz derives an encryption key from the recovery phrase with HKDF-SHA256 and
encrypts each alias payload with XChaCha20-Poly1305. Each record uses a fresh
random nonce. The server stores only the encrypted blob for the alias payload.

Because encryption happens before upload, the sync service should not be able to
read alias names or commands.

## What the server can see

The sync server still needs some information to operate. It can see:

- Username.
- User ID.
- Password hash.
- Session token hashes.
- Record IDs.
- Record type.
- Record versions.
- Record update timestamps.
- Encrypted record blobs.
- Rate-limit counters.
- Request metadata normally visible to the hosting platform.

This metadata can reveal that an account exists, how many records it has, and
roughly when records changed. It should not reveal the plaintext aliases.

## If the sync database is compromised

If an attacker gains read access to the sync database, they should see encrypted
records and metadata, not plaintext aliases.

They should not be able to decrypt alias names or commands unless they also have
the user's recovery phrase or access to a client machine where that phrase is
available.

If an attacker gains write access to the database, they may be able to delete
records, withhold records, replay older encrypted records, or corrupt encrypted
data. XChaCha20-Poly1305 should prevent them from creating new valid alias
payloads without the encryption key, but it does not stop denial-of-service or
rollback-style attacks by a malicious storage layer.

## Local storage

Aliaz stores the local alias database on disk. Local aliases are readable by
processes or users that can access that database file. The managed shell alias
files generated by `aliaz init` are also plaintext, because shells need to read
them directly.

The recovery phrase is stored in the operating system credential store when
available. On systems without a credential service, Aliaz falls back to a
private file-backed secret store under the config directory. Set
`ALIAZ_SECRET_HOME` to choose the file-backed store location explicitly. The
security of that secret depends on the security of the local account, device,
credential store, and any configured secret-store directory.

For maximum privacy on shared or managed machines:

- Prefer interactive password and recovery phrase prompts.
- Avoid passing secrets as command-line arguments.
- Protect local backups that include Aliaz data or generated shell files.
- Lock down access to the operating system account, credential store, and
  `ALIAZ_SECRET_HOME` if configured.

## Hosted sync and custom servers

By default, Aliaz uses the hosted Cloudflare Worker sync API. You can point the
CLI at another compatible server:

```sh
aliaz register --username ada --sync-url https://sync.example.com
aliaz login --username ada --sync-url https://sync.example.com
```

Using a custom server changes who operates the metadata and encrypted blob
storage. It does not change the client-side encryption model.

## Current limitations

Aliaz is built to be privacy-preserving, but careful users should know the
boundaries:

- The project has not had a formal third-party security audit.
- Local alias data and generated shell files are not encrypted at rest by Aliaz.
- Metadata is visible to the sync service.
- A malicious or compromised server can deny service, delete data, corrupt data,
  or replay older encrypted records.
- A compromised client machine can expose aliases and the recovery phrase.
- Losing the recovery phrase means losing the ability to decrypt synced aliases.

## Practical rule

Treat the recovery phrase like the key to your aliases. The sync server can
store encrypted data for you, but the recovery phrase is what makes that data
readable.

If someone only gets the sync database, they should not be able to read your
aliases. If someone gets your recovery phrase or controls your client machine,
they can.
